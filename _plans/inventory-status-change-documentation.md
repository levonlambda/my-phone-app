# Inventory Status Change — Firestore Documentation

Reference document for implementing inventory status changes in a mobile app. This describes the exact data model and operations the web app performs when a user changes an inventory item's status.

---

## 1. Firestore Collections Involved

Only **two collections** are affected when a status changes. No ledger entries, no cascading updates to any other collection.

### 1.1 `inventory` Collection

Each document represents a single physical phone unit.

| Field          | Type     | Description                                      | Example                |
|----------------|----------|--------------------------------------------------|------------------------|
| `manufacturer` | string   | Phone manufacturer                               | `"Samsung"`            |
| `model`        | string   | Phone model name                                 | `"Galaxy S24 Ultra"`   |
| `ram`          | string   | RAM size                                         | `"12GB"`               |
| `storage`      | string   | Storage capacity                                 | `"256GB"`              |
| `color`        | string   | Color variant                                    | `"Titanium Black"`     |
| `imei1`        | string   | Primary IMEI (15 digits), can be empty            | `"356789012345678"`    |
| `imei2`        | string   | Secondary IMEI (15 digits), optional              | `"356789012345679"`    |
| `serialNumber` | string   | Serial number, alternative to IMEI                | `"ABC123XYZ"`          |
| `barcode`      | string   | Product barcode                                  | `"8901234567890"`      |
| `dealersPrice` | number   | Wholesale/dealer cost                             | `25000`                |
| `retailPrice`  | number   | Selling/retail price                              | `30000`                |
| `location`     | string   | Physical storage location                         | `"Main Store"`         |
| `supplier`     | string   | Supplier document ID (references `suppliers`)     | `"abc123def456"`       |
| `supplierId`   | string   | Duplicate supplier ID field (legacy/compat)       | `"abc123def456"`       |
| `procurementId`| string   | Links to a procurement order                      | `"proc_001"`           |
| `status`       | string   | Current inventory status (see Section 2)          | `"On-Hand"`            |
| `dateAdded`    | string   | Date item was first added (M/D/YYYY)              | `"3/15/2026"`          |
| `lastUpdated`  | string   | Date of last modification (M/D/YYYY)              | `"3/21/2026"`          |
| `isArchived`   | boolean  | Whether item has been archived                    | `false`                |
| `archiveBatch` | string   | Batch identifier if archived                      | `"batch_20260301"`     |

**Document ID:** Auto-generated by Firestore (`addDoc`).

### 1.2 `inventory_counts` Collection

Each document is an aggregate counter for a specific phone configuration (manufacturer + model + RAM + storage + color).

| Field          | Type     | Description                                      | Example                |
|----------------|----------|--------------------------------------------------|------------------------|
| `manufacturer` | string   | Phone manufacturer                               | `"Samsung"`            |
| `model`        | string   | Phone model name                                 | `"Galaxy S24 Ultra"`   |
| `ram`          | string   | RAM size                                         | `"12GB"`               |
| `storage`      | string   | Storage capacity                                 | `"256GB"`              |
| `color`        | string   | Color variant                                    | `"Titanium Black"`     |
| `total`        | number   | Total items of this configuration                 | `10`                   |
| `onHand`       | number   | Count of items with status "On-Hand"              | `5`                    |
| `onDisplay`    | number   | Count of items with status "On-Display"           | `2`                    |
| `sold`         | number   | Count of items with status "Sold"                 | `1`                    |
| `reserved`     | number   | Count of items with status "Reserved"             | `1`                    |
| `defective`    | number   | Count of items with status "Defective"            | `1`                    |
| `lastUpdated`  | string   | Date of last count update (M/D/YYYY)              | `"3/21/2026"`          |

**Document ID:** Deterministic, computed from the phone specs (see Section 3).

## 2. Status Values

### 2.1 The 5 Statuses

| Stored Value in Firestore | Display Label in UI | `inventory_counts` Field |
|---------------------------|---------------------|--------------------------|
| `"On-Hand"`               | `"Stock"`           | `onHand`                 |
| `"On-Display"`            | `"On-Display"`      | `onDisplay`              |
| `"Sold"`                  | `"Sold"`            | `sold`                   |
| `"Reserved"`              | `"Reserved"`        | `reserved`               |
| `"Defective"`             | `"Defective"`       | `defective`              |

### 2.2 Status-to-Field Mapping Logic

When mapping a status string to its corresponding `inventory_counts` field name:

```
"On-Hand"    → "onHand"
"On-Display" → "onDisplay"
"Sold"       → "sold"
"Reserved"   → "reserved"
"Defective"  → "defective"
```

The web app uses a chain of ternary expressions for this mapping:

```javascript
const fieldName =
  status === 'On-Hand'    ? 'onHand'    :
  status === 'On-Display' ? 'onDisplay' :
  status === 'Sold'       ? 'sold'      :
  status === 'Reserved'   ? 'reserved'  :
  'defective';  // fallback/default
```

### 2.3 Default Status

When an inventory item has no status set, it defaults to `"On-Hand"`:
```javascript
status: data.status || 'On-Hand'
```

## 3. Inventory Counts Document ID

The `inventory_counts` document ID is computed deterministically from 5 fields:

### Formula

```
{manufacturer}_{model}_{ram}_{storage}_{color}
```

All values are **lowercased** and **spaces are replaced with underscores**.

### Implementation

```javascript
function createInventoryId(manufacturer, model, ram, storage, color) {
  return `${manufacturer}_${model}_${ram}_${storage}_${color}`
    .replace(/\s+/g, '_')
    .toLowerCase();
}
```

### Example

For a phone with:
- manufacturer: `"Samsung"`
- model: `"Galaxy S24 Ultra"`
- ram: `"12GB"`
- storage: `"256GB"`
- color: `"Titanium Black"`

The document ID would be:
```
samsung_galaxy_s24_ultra_12gb_256gb_titanium_black
```

## 4. Date Format

### `getCurrentDate()` Function

Returns the current date as a string in `M/D/YYYY` format (no leading zeros):

```javascript
function getCurrentDate() {
  const today = new Date();
  const month = today.getMonth() + 1; // zero-based, so +1
  const day = today.getDate();
  const year = today.getFullYear();
  return `${month}/${day}/${year}`;
}
```

### Examples
- March 5, 2026 → `"3/5/2026"` (not `"03/05/2026"`)
- December 25, 2026 → `"12/25/2026"`

### Important Note

The web app has an inconsistency: in the inline edit flow (`InventoryTable.jsx`), the `inventory_counts` document uses `new Date().toLocaleDateString()` instead of `getCurrentDate()`. The output depends on the browser's locale but typically produces the same `M/D/YYYY` format in US English locales. For the mobile app, use the `getCurrentDate()` format (`M/D/YYYY` without leading zeros) for consistency.

## 5. Status Change Process — Step by Step

When a user changes an item's status (e.g., from "On-Hand" to "Sold"), the following operations occur **in this exact order**:

### Step 1: Build the Update Object

Construct the data to write to the `inventory` document:

```javascript
updateData = {
  manufacturer: editFormData.manufacturer,
  model: editFormData.model,
  ram: editFormData.ram,
  storage: editFormData.storage,
  color: editFormData.color,
  imei1: editFormData.imei1,
  serialNumber: editFormData.serialNumber || '',
  barcode: originalItem.barcode || '',       // preserved from original
  status: editFormData.status,               // THE NEW STATUS
  location: originalItem.location || '',     // preserved from original
  supplier: editFormData.supplier || '',
  lastUpdated: getCurrentDate()              // e.g. "3/21/2026"
}
```

### Step 2: Update the `inventory` Document

> **In plain terms:** Each physical phone has its own document in the `inventory` collection (with a Firestore auto-generated ID like `"xYz123AbC"`). This step simply opens that one document and overwrites its fields with the new values — most importantly the `status` field (e.g., changing it from `"On-Hand"` to `"Sold"`) and the `lastUpdated` field (stamped with today's date). This is a straightforward, single-document write — it is NOT wrapped in a transaction. All other editable fields (manufacturer, model, imei1, etc.) are also re-saved at the same time, even if they didn't change.

Direct update (not inside a transaction):

```
Collection: "inventory"
Document ID: <the item's Firestore document ID>
Operation: updateDoc(itemRef, updateData)
```

This writes all fields from the update object, including the new `status` and the updated `lastUpdated`.

### Step 3: Check If Status Actually Changed

Compare the original status to the new status:

```javascript
if (originalItem.status !== editFormData.status) {
  // proceed to Step 4
}
```

If the status did NOT change, skip Steps 4-5 entirely — no `inventory_counts` update needed.

### Step 4: Compute the `inventory_counts` Document ID

> **In plain terms:** Unlike the `inventory` collection (where each physical phone has its own document), the `inventory_counts` collection has **one shared counter document per phone configuration**. For example, if you have 10 Samsung Galaxy S24 Ultra 12GB/256GB Titanium Black units, all 10 share a single counter document that tracks how many are in each status.
>
> To find the right counter document, you build its ID by joining the 5 spec fields with underscores and lowercasing everything. Your mobile app must compute this exact same string to update the correct counter.
>
> **Example:** A phone with manufacturer `"Samsung"`, model `"Galaxy S24 Ultra"`, ram `"12GB"`, storage `"256GB"`, color `"Titanium Black"` produces the document ID:
> `samsung_galaxy_s24_ultra_12gb_256gb_titanium_black`

```javascript
const inventoryId = createInventoryId(
  editFormData.manufacturer,
  editFormData.model,
  editFormData.ram,
  editFormData.storage,
  editFormData.color
);
```

This produces the document ID in the `inventory_counts` collection (see Section 3).

### Step 5: Run a Transaction to Update `inventory_counts`

> **In plain terms:** The counter document tracks how many items of a specific phone configuration are in each status. When you change one phone from "On-Hand" to "Sold", you need to subtract 1 from the `onHand` counter and add 1 to the `sold` counter. This runs inside a **Firestore transaction** because two users (or the web app and mobile app) could change items of the same configuration at the same time — the transaction guarantees the counters stay accurate and no increments/decrements are lost.
>
> **Concrete example:** Say the counter document currently looks like this:
> ```
> { total: 10, onHand: 5, onDisplay: 2, sold: 1, reserved: 1, defective: 1 }
> ```
> You change one item from `"On-Hand"` → `"Sold"`. After the transaction:
> ```
> { total: 10, onHand: 4, onDisplay: 2, sold: 2, reserved: 1, defective: 1 }
> ```
> Notice `total` stays at 10 — it never changes during a status update. Only the two affected status fields change by 1 each. The `total` field only changes when items are added to or deleted from inventory.
>
> If the counter document doesn't exist (unusual edge case), the transaction simply does nothing and doesn't throw an error.

This is the critical step. It runs inside a **Firestore transaction** to ensure atomicity:

```
Collection: "inventory_counts"
Document ID: <computed inventoryId from Step 4>
```

**Inside the transaction:**

1. **Read** the `inventory_counts` document
2. **Check** if the document exists
3. If it exists:
   a. **Determine the decrement field** — map the OLD status to its field name (e.g., `"On-Hand"` → `"onHand"`)
   b. **Determine the increment field** — map the NEW status to its field name (e.g., `"Sold"` → `"sold"`)
   c. **Update** the document with:
      - `[decrementField]`: `increment(-1)` (Firestore atomic decrement)
      - `[incrementField]`: `increment(1)` (Firestore atomic increment)
      - `lastUpdated`: current date string
4. If the document does NOT exist: **no action is taken** (the transaction silently skips)

**Note:** The `total` field is NOT modified during a status change — it only changes when items are added or deleted.

### Pseudocode Summary

```
// Step 2: Update the inventory item
firestore.collection("inventory").doc(itemId).update({
  ...updateData,
  status: newStatus,
  lastUpdated: "M/D/YYYY"
});

// Step 3-5: If status changed, update counts
if (oldStatus !== newStatus) {
  inventoryId = createInventoryId(manufacturer, model, ram, storage, color);

  runTransaction {
    countsDoc = read("inventory_counts", inventoryId);
    if (countsDoc.exists) {
      oldField = statusToFieldName(oldStatus);   // e.g. "onHand"
      newField = statusToFieldName(newStatus);    // e.g. "sold"

      update("inventory_counts", inventoryId, {
        [oldField]: increment(-1),
        [newField]: increment(1),
        lastUpdated: "M/D/YYYY"
      });
    }
  }
}
```

## 6. Alternative Status Change Path (Full Edit Form)

The web app has a second code path for status changes via the "Edit Full Details" form. This uses `updatePhoneInInventory()` from `InventoryService.js`. The logic is the same with one difference:

**It handles the case where the phone's specs (manufacturer, model, ram, storage, color) also changed**, meaning the old and new `inventory_counts` documents might be different:

```
oldInventoryId = createInventoryId(originalData.manufacturer, originalData.model, ...)
newInventoryId = createInventoryId(phoneData.manufacturer, phoneData.model, ...)
```

Inside the transaction:
1. Read BOTH the old and new `inventory_counts` documents
2. Decrement the old status field on the OLD `inventory_counts` document
3. Increment the new status field on the NEW `inventory_counts` document

For a mobile app that only changes status (not phone specs), this distinction is unlikely to matter, but it's documented here for completeness.

## 7. Edge Cases and Constraints

### 7.1 Archived Items Are Read-Only
Items with `isArchived: true` cannot be edited or deleted in the web UI. The edit/delete buttons are hidden and replaced with "Archived - Read Only" text. The mobile app should enforce the same constraint.

### 7.2 No Validation on Status Change
The inline edit does NOT validate IMEI uniqueness or any other field when saving. It directly writes the update. Only the full edit form (`InventoryService.js`) performs duplicate IMEI checks.

### 7.3 Missing `inventory_counts` Document
If the `inventory_counts` document for the item's spec configuration doesn't exist, the transaction simply does nothing — no error is thrown, and no document is created. Counts are only initialized when a brand-new item is first added to inventory.

### 7.4 No Side Effects to Other Collections
A status change does NOT:
- Create or update any `supplier_ledger` entries
- Modify the `suppliers` collection
- Modify the `procurements` or `procurement_items` collections
- Trigger any archiving
- Send any notifications

### 7.5 `total` Count Is Not Modified
During a status change, only the individual status counter fields change. The `total` field on `inventory_counts` remains unchanged. It only changes when items are added (`total: increment(1)`) or deleted (`total: increment(-1)`).

### 7.6 Concurrent Access
The `inventory_counts` update uses a Firestore transaction, which provides optimistic concurrency control. If two status changes on items with the same spec happen simultaneously, Firestore will retry the transaction automatically. The `inventory` document update itself is a direct `updateDoc` (not transactional), so last-write-wins applies to the item document.

## 8. Mobile App Implementation Checklist

To replicate this behavior in a mobile app:

1. **Read the current item** from `inventory/{itemId}` to get the original status
2. **Update the item** in `inventory/{itemId}` with the new status and `lastUpdated` set to current date in `M/D/YYYY` format (no leading zeros)
3. **If status changed**, compute the `inventory_counts` document ID using `createInventoryId()`
4. **Run a transaction** on `inventory_counts/{inventoryId}`:
   - Read the document
   - If it exists: decrement old status field by 1, increment new status field by 1, update `lastUpdated`
   - If it doesn't exist: do nothing
5. **Respect archived items** — do not allow status changes on items where `isArchived === true`

### Firebase SDK Methods Needed
- `updateDoc()` — for the inventory item update
- `runTransaction()` — for the atomic counter update
- `increment()` — Firestore's `FieldValue.increment()` for atomic counter operations
- `doc()` — to create document references
- `getDoc()` or `transaction.get()` — to read documents
